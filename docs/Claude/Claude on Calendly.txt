**Calendly, nitpicked**

The calendar sync isn't real sync, it's a fragile poll-and-hope layer with no diagnostics. Users report a linked calendar correctly marking events as busy while Calendly still shows that time as open, with Office 365 sync silently broken despite being advertised, and support's only troubleshooting step being to double-check the event says "busy" rather than "free busy" — no actual trace of why the system thinks what it thinks, just a script. When sync drifts, there's no way for the user to see the decision; the gap calendar-sync products always have when "sync" means "remember to repoll" instead of "react to a push event."

It has an actual integrity bug, not just a UX gripe. On its own community forum, an admin reported that a single one-on-one event type let two different clients book the exact same time slot, with confirmation emails roughly ten hours apart, recurring multiple times in one week. A one-on-one event type claiming a slot should be structurally incapable of double-assignment — that it happened at all means the claim isn't an atomic transition, it's a write that can race.

The data model conflates "the calendar owner" with "the bookable thing," and it shows constantly. Calendly's whole architecture treats an account as one person who can't be in two places, so the moment you have multiple interchangeable resources — five workstations that should each be independently bookable, or three town assessors who should each take the same 9:10 slot with a different resident — you're stuck either paying for fake "seats" that aren't people, or using Group events that lose per-resource identity, or a Round Robin flow where a resource has to manually "accept an invitation" before it's bookable. That's not a missing feature, it's the foundational data model not generalizing past "one human's personal calendar."

Timezone handling is a running joke in its own reviews. Multiple reviewers separately describe genuine confusion about which timezone a booking actually landed in, in one case a slot set for 8:30am Pacific generating a confirmation email in UTC instead, with the support chat disconnecting mid-troubleshooting. For a product whose entire job is "tell two people the same unambiguous time," that's about as close to a core-value-prop failure as it gets.

And the pricing model fights the use case you actually care about: basic customization is locked behind higher pricing tiers even for the single-person, single-event-type case — the exact "just let me see open slots and pick one" scenario that should cost Calendly approximately nothing to serve for free, forever.

**The greenfield version**

Three changes fix essentially everything above, and all three are things Dominatus already does for free.

First, the bookable resource is the atomic entity, not the calendar owner. A provider can expose N resources (chairs, rooms, staff, "any of these three interchangeable slots"), each independently schedulable, with no seat-per-person billing hack and no Group-event identity loss — this alone kills the workstation/multi-assessor problem outright.

Second, claiming a slot is a typed state transition with a real lock, not a write-and-hope. Below is the lifecycle: open, then a short atomic hold while the booker finishes entering their info, then confirmed, branching to completed or cancelled. Two people tapping the same slot within the same second isn't a race condition to debug in production, it's a transition that's structurally exclusive by construction — exactly your HFSM-plus-blackboard pattern, just pointed at a calendar instead of an NPC.Third, the calendar a provider already uses stays the single source of truth — never duplicated, never drifting. Sync is push-based where the provider supports it, and every sync event gets logged the way your cassette replay logs every `Llm.Decide` call, so "why does it think I'm free" has an actual inspectable answer instead of a support agent asking you to re-check a checkbox.

Everything else follows from those three: booking itself is phone-number-first, no account, no password, text the slot you want and get a one-tap confirm back — the SMS-native flow Calendly was never built for because it grew out of sales-call links shared by email. Timezone gets resolved once, server-side, and restated explicitly in the confirmation text rather than trusted to whatever the recipient's email client decides to render. And the single-resource, single-event-type case — a solo doctor, a solo hairdresser, exactly what you described two messages ago — stays free forever, because the marginal cost of hosting someone's open slots is close to zero; you monetize the multi-resource tier and optional bolt-ons (a Stripe deposit hook for the no-show-prone businesses, nothing forced on anyone who doesn't need it) instead of fencing off basic functionality the way Calendly does.

None of this requires an LLM anywhere in the critical path. It's a five-state machine, a calendar webhook, and an SMS provider — which is exactly the part of your stack that's already finished.
